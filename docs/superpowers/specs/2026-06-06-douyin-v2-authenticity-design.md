# GitHub Video Maker V2 - 抖音可发布性优化设计

## 背景

当前项目已经能从 GitHub 仓库生成视频，但成片不适合直接发布到抖音。主要问题不是单点 bug，而是生产范式偏“自动演示模板”：全程 GitHub 页面、模板化口播、横屏构图、假鼠标操作和一次性生成，导致画面、配音、文案整体都显得不真实。

V2 的目标不是立刻做到全自动爆款，而是把系统改成“可审核的短视频生产流水线”。每一步都要产出中间文件，让人能判断、修改、复用。

## 预设条件

- 第一阶段优先服务中文抖音/短视频账号，不兼顾 YouTube 横屏长视频。
- 第一阶段默认输出竖屏 `1080x1920`。
- 第一阶段保留 `edge-tts`，但先解决文案和节奏问题，不把换音色当成核心修复。
- 第一阶段不做自动发布、不做账号运营、不做热点抓取。
- 第一阶段不保证所有 GitHub 项目都适合生成视频；不适合的项目应该被明确拒绝或降级。

## 核心原则

1. 先判断值不值得做，再生成视频。
2. 先找真实证据，再写口播。
3. GitHub 页面只作为来源证明，不作为主要画面。
4. 口播禁止空泛夸赞，每句话必须绑定问题、证据或判断。
5. 一键生成之前必须有可审核中间产物。

## 新流水线

```text
GitHub URL
  -> repo_info.json
  -> creative_brief.json
  -> asset_manifest.json
  -> shot_plan.json
  -> script.json
  -> audio/
  -> preview_frames/
  -> final.mp4
```

## 模块设计

### 1. Creative Brief 选题层

职责：判断这个项目是否值得做成短视频，并给后续脚本提供内容方向。

输入：

- 仓库名称、描述、stars、topics、README 摘要
- README 中的图片、链接、安装说明、功能列表

输出 `creative_brief.json`：

- `target_audience`: 目标观众
- `viewer_pain`: 观众痛点
- `one_line_value`: 一句话价值
- `proof_points`: 3-5 个可证明卖点
- `visual_opportunities`: 可视化素材机会
- `risks`: 假感/误导/素材不足风险
- `recommendation`: `produce` 或 `skip`
- `reason`: 推荐原因

验收标准：

- README 没有可视化素材、功能过泛或无法解释给普通观众时，允许输出 `skip`。
- 每个 `proof_point` 必须来自仓库描述、README 或可见素材，不能凭空夸。
- 不再默认所有项目都是“宝藏项目”。

### 2. Asset Manifest 素材层

职责：从 GitHub/README/项目官网中整理可用画面素材。

第一阶段素材类型：

- README 图片
- README 里的项目官网链接截图
- GitHub 仓库首页截图
- README 关键代码块截图
- README 功能段落截图

输出 `asset_manifest.json`：

- `assets[]`
- 每个素材包含 `type`、`source`、`path`、`caption`、`use_case`、`quality`

验收标准：

- 成片主体画面不能只来自 GitHub 仓库首页。
- 如果只有 GitHub 页面可用，系统要在 brief 中标记低质量风险。
- 素材采集失败时给出明确原因，不静默用模板画面顶上。

### 3. Shot Plan 分镜层

职责：把 brief 和素材变成竖屏短视频结构。

推荐结构：

- 0-3 秒：具体痛点或反常识开头
- 3-8 秒：项目一句话价值
- 8-22 秒：2-3 个证据镜头
- 22-28 秒：适合谁/不适合谁
- 28-32 秒：收藏或 Star CTA

输出 `shot_plan.json`：

- `shots[]`
- 每个镜头包含 `start`、`duration`、`visual_asset`、`visual_treatment`、`narration_intent`、`subtitle`

验收标准：

- 每个镜头都有明确视觉素材，不能只有抽象描述。
- 至少 70% 主体镜头不是 GitHub 首页全景。
- 每个镜头时长由口播长度和画面可读性决定，不再只用脚本字段硬切。

### 4. Script 文案层

职责：从“夸项目”改成“帮观众判断项目是否有用”。

文案规则：

- 禁止默认使用“兄弟们”“卧槽”“绝了”“神器”“宝藏项目”“真的牛”。
- 禁止没有证据的强推荐。
- 优先使用具体场景：谁遇到什么问题，这个项目怎么解决。
- 允许保留主观判断，但必须说明判断依据。
- CTA 从“快去 Star”改为“适合就收藏/去试试/看源码”。

验收标准：

- 口播稿去掉夸张词后仍然成立。
- 每段口播都能回指一个 proof point 或视觉素材。
- 单句不只追求短，还要完整表达一个意思。

### 5. Vertical Composer 竖屏合成层

职责：让素材适配抖音竖屏观看。

画面结构：

- 顶部：短标题或问题
- 中部：主素材裁切/放大/平移
- 底部：字幕安全区

第一阶段改动：

- 输出 `1080x1920`
- 横屏截图自动放大裁切到中部主画面
- 字幕不遮挡核心内容
- 去掉默认粒子效果和大红框滥用

验收标准：

- 手机竖屏观看时，核心文字不用暂停也能看清。
- 字幕不压住网页/产品截图的关键区域。
- 成片不再出现横屏视频直接塞进抖音画面的效果。

### 6. Review Gate 审核层

职责：避免一键到底造成不可控假感。

第一阶段新增 CLI 选项：

- `--dry-run`: 只生成 `creative_brief.json`、`asset_manifest.json`、`shot_plan.json`、`script.json`
- `--from-plan`: 从已有分镜继续生成视频
- `--vertical`: 输出竖屏视频

验收标准：

- 用户可以在视频生成前修改 brief、分镜和口播。
- 重新生成视频时不必重新抓取所有仓库信息。
- 中间产物能独立阅读和人工审稿。

## 非目标

- 不做抖音自动发布。
- 不做爆款标题库。
- 不做复杂视频编辑器 UI。
- 不做真人克隆音色。
- 不做对所有仓库的一键保证。

## 开发顺序

1. 新增 `creative_brief` 生成模块和数据模型。
2. 新增 `asset_manifest` 生成模块，先支持 README 图片、官网链接、关键段落截图。
3. 新增 `shot_plan` 生成模块，把 brief 和素材映射成分镜。
4. 重写脚本 prompt，让口播绑定 proof points。
5. 修改 composer 支持真实竖屏输出。
6. 增加 `--dry-run` 和 `--from-plan`。
7. 用现有两个输出项目做回归样例。

## 验收样例

以 `zyronon/TypeWords` 为例，V2 成片应该满足：

- 开头说清楚“背单词只认识不会用”的具体问题，而不是泛称“学英语神器”。
- 主体展示 README 里的产品截图或官网截图，不只展示 GitHub 文件列表。
- 口播说明“跟打、听写、课文练习”分别解决什么问题。
- 结尾说明适合英语学习者/不适合只想刷题的人，而不是只喊 Star。
- 输出为 `1080x1920` 竖屏。

以 `simonlin1212/a-stock-data` 为例，V2 成片应该满足：

- 开头说清楚“A 股数据源分散、接口不稳定”的具体问题。
- 主体展示数据源、端点、调用示例或返回结果。
- 口播避免“全都有”这种不可验证说法，改成具体覆盖范围。
- 如果没有足够可视化素材，brief 应提示适合做“技术讲解”而不是“产品演示”。

## 风险

- 项目官网截图可能需要额外浏览器抓取和失败处理。
- README 图片质量参差不齐，部分项目素材不足。
- 竖屏裁切需要为不同素材做安全区判断。
- 真实感提升后，自动化程度会下降，需要接受人工审核步骤。

## 最小可交付版本

最小 V2 不追求最终完美成片，只要求：

- 能生成 `creative_brief.json`
- 能生成 `asset_manifest.json`
- 能生成 `shot_plan.json`
- 能生成更克制的 `script.json`
- 能输出一个竖屏 `final.mp4`
- 能通过 `--dry-run` 停在审核阶段

只要这六项成立，项目就从“模板视频生成器”进入“可迭代的视频生产工具”阶段。
