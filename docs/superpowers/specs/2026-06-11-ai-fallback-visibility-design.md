# 2026-06-11 AI 生成与回退标识设计

## 背景

控制台已经会在模型不可用时回退到启发式候选分析、默认口播模板或跳过质检，但这些状态散落在日志里，用户不能快速判断当前内容到底来自 AI、缓存还是规则回退。

## 本次范围

只补齐 `P1-1` 要求的最小可见性：

1. 后端为任务补一个 `candidate_source` 摘要，统一记录候选数据来源、候选分析来源和排序来源。
2. 候选区显示本轮候选来源，明确区分 `AI / 启发式 / 缓存`。
3. 口播区显示 `AI 成功 / 模型跳过回退 / AI 失败后回退`。
4. 产物摘要区显示最新模型调用状态和当前口播来源。

## 设计取舍

- 不做逐字段来源追踪。
  - 原因：当前验收只要求用户能快速识别回退状态，不需要把每个描述字段都拆成 provenance。
- 不新增独立接口。
  - 原因：现有 `job_detail` 已经返回任务详情，直接在任务快照里补来源摘要即可。
- 失败原因只展示短摘要。
  - 原因：详细错误仍保留在日志和 diagnostics，摘要区只负责快速判断。

## 验收点

1. 不配置模型时，候选区和口播区都能明确显示回退状态。
2. 模型调用失败时，UI 能看到失败后的回退说明和短错误原因。
3. 产物摘要能看到最近一次模型调用状态，不阻塞用户继续手改脚本或继续流程。
