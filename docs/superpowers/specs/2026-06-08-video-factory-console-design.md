# 视频工厂控制台设计文档

## 目标

为现有 GitHub 视频生成工厂建设一个本机单人使用的可视化控制台。第一版要能管理配置、创建 GitHub 热榜草稿任务、让用户确认项目和口播脚本，然后复用现有竖屏视频流水线生成带编号的最终视频。

这套控制台会作为后续更多视频生产线的底座，但第一条生产可用的完整流程只做 GitHub 热榜竖屏视频。

## 产品范围

### 第一版包含

- 本机浏览器控制台，单人使用。
- 单页工作台，风格参考 Claude Code：信息密集、深色、克制、状态优先、偏生产工具。
- GitHub 热榜竖屏视频作为第一版唯一完整接入的生产线。
- 通过 GitHub Search API 按日榜、周榜、月榜拉取候选项目。
- GitHub Token 可选配置，但界面推荐配置以提升稳定性。
- AI 分析候选项目，然后由用户手动选择和排序。
- AI 生成口播脚本，然后由用户手动编辑和确认。
- 按任务类型配置模型，例如候选分析、排序、标题钩子、口播、润色、质检。
- 支持 OpenAI、Anthropic、DeepSeek、小米，以及国内常见 OpenAI-compatible 服务。
- 固定模板结构 + 可调风格参数。
- 后台分阶段执行任务，显示进度、日志、失败原因，并在可行时支持从失败阶段继续。
- 输出文件和任务目录使用稳定编号。

### 第一版不包含

- 多用户账号、权限或计费。
- 云端部署或 SaaS 化。
- 完全无人值守自动发布。
- 不经用户确认就全自动生成最终视频。
- 可视化拖拽时间线或分镜编辑器。
- 单个 GitHub 项目推荐的完整生产闭环。
- 科普类视频的完整生产闭环。
- 横屏视频的完整打磨。

## 第一版产品决策

- 先做任务控制台，不做完整视频编辑器。
- 复用当前 Python 视频流水线，不重写视频生成核心。
- 架构面向未来多生产线，但第一版只把 GitHub 热榜接到端到端可用。
- 支持手动或定时生成草稿，但最终渲染前必须由用户确认。
- 视频风格必须确定。AI 只负责填充模板内容槽位，不在运行时临场设计布局。

## 系统架构

```text
浏览器工作台
  -> 本地后端 API
  -> 任务执行器
  -> 流水线适配层
  -> 现有 Python 视频流水线
  -> output 任务目录和最终 mp4
```

### Web 控制台

控制台在一个工作台内展示配置、候选项目审查、口播编辑、任务进度、日志和输出文件。它应该像一个开发者生产工具，而不是营销型后台页面。

### 本地后端 API

后端负责保存本地配置、创建任务、读取任务状态、流式或轮询读取日志、列出历史任务、暴露输出产物。第一版是本机单人使用，因此不做登录认证。

### 任务执行器

任务执行器负责分阶段执行和持久化状态。它追踪这些状态：

- `draft_pending`
- `collecting_candidates`
- `analyzing_candidates`
- `awaiting_project_confirmation`
- `generating_script`
- `awaiting_script_confirmation`
- `preparing_plan`
- `capturing_assets`
- `generating_tts`
- `composing_video`
- `post_processing`
- `completed`
- `failed`

每个阶段都写入日志，并保存足够的中间状态以便检查结果。阶段失败时需要显示失败阶段、相关供应商和模型（如果有关）、简明错误摘要。

### 流水线适配层

适配层把用户确认后的热榜数据转换成当前流水线需要的文件：

- `info.json`
- `asset_manifest.json`
- `shot_plan.json`
- `script.json`

现有流水线可以继续生成 `final.mp4`。任务完成后，控制台再生成带编号的正式输出文件名。

## GitHub 热榜工作流

### 草稿阶段

1. 用户选择 `daily`、`weekly` 或 `monthly`。
2. 系统调用 GitHub Search API 拉取近期候选仓库。
3. 系统保存候选快照，避免后续 GitHub 数据变化影响该任务。
4. AI 分析每个候选项目，输出：
   - 一句话价值
   - 推荐理由
   - 风险提示
   - 适合人群
   - 画面可视化潜力
   - 内容评分
5. 控制台展示 20-30 个候选项目。
6. 用户选择并排序 5 个或 10 个项目。
7. AI 生成分段热榜口播脚本。
8. 用户编辑并确认口播脚本。

### 出片阶段

1. 系统写入已确认的任务快照。
2. 适配层生成分镜、素材清单和脚本文件。
3. 系统采集 GitHub 页面和 README 素材。
4. 系统生成 TTS 音频。
5. 竖屏 composer 渲染视频。
6. 执行 BGM 和后处理。
7. 系统保存日志、预览帧、JSON 文件、`final.mp4` 和带编号的正式视频文件。

## 输出命名

最终视频使用稳定编号文件名：

```text
GH-HOTLIST-YYYYMMDD-###-安全标题.mp4
```

示例：

```text
GH-HOTLIST-20260608-001-本周AI开源项目热榜.mp4
GH-HOTLIST-20260608-002-GitHub日榜10个宝藏项目.mp4
```

任务目录使用同一个编号：

```text
output/jobs/GH-HOTLIST-20260608-001/
  task.json
  candidates.json
  selected_projects.json
  script.json
  shot_plan.json
  asset_manifest.json
  logs.txt
  preview_frames/
  final.mp4
  GH-HOTLIST-20260608-001-本周AI开源项目热榜.mp4
```

每日编号按生产线前缀和日期递增。

## 模型配置

### 供应商

第一版支持这些供应商类型：

- OpenAI
- Anthropic
- DeepSeek
- 小米
- 其他 OpenAI-compatible 服务

每个供应商配置保存：

- 显示名称
- 供应商类型
- API Key
- Base URL（适用时）
- 默认模型
- 是否启用
- 最近一次连接测试结果

国内供应商和 OpenAI-compatible 服务必须把 `base_url` 作为一等配置项，而不是写死。

### 任务路由

用户可以按任务类型指定模型：

- 候选项目分析
- 热榜排序和筛选
- 标题与钩子生成
- 口播脚本生成
- 脚本润色
- 事实与夸大检查

第一版应内置默认路由方案，让用户不必一开始就手动配置每个任务。

API Key 只保存在本地供应商配置里。任务快照只记录供应商名称、模型名称、任务类型和调用时间，不记录 API Key。

## 模板与风格系统

第一版使用确定性的模板。AI 可以填充内容槽位，也可以建议文案，但不能改变分镜结构或布局规则。

### 生产线模板

`github_hotlist_vertical_v1` 定义：

- 开场钩子
- 榜单总览
- 重复项目排名卡片
- 趋势总结
- 结尾互动语

### 可调参数

第一版只暴露高价值参数：

- 项目数量：5 或 10
- 风格：科技深色、极简白底、黑金强调
- 字幕模式：大字强钩子、标准解释型
- BGM：默认、无、自定义本地文件
- 口播语气：专业测评、爆款短视频、冷静解读
- 输出方向：第一版只支持竖屏，横屏预留

### 内容槽位

AI 填充这些槽位：

- 开场钩子
- 榜单标题
- 单项目钩子
- 解决了什么
- 技术亮点
- 适合谁
- 口播段落
- 结尾语

新增视频风格时，应固化成带版本号的模板，例如 `github_hotlist_vertical_v2`，而不是每次让 AI 自由重新生成风格。

## UI 设计

界面打开后直接进入工作台，不做 landing page。

### 布局

- 顶部任务栏：生产线、时间维度、模板、主操作按钮、当前任务编号和状态。
- 左侧配置栏：GitHub Token 状态、模型路由、模板参数、输出设置。
- 中间工作区：根据任务阶段展示候选项目表、口播编辑器、阶段进度和日志。
- 右侧产物栏：任务摘要、预览帧、最终视频、历史任务、打开输出目录、重跑操作。
- 底部日志抽屉：可展开的实时日志区域，保留命令行式透明感。

### 视觉风格

- 深灰分层背景。
- 紧凑表格和面板。
- 细边框和分隔线。
- 任务编号、模型名、路径、日志使用 monospace。
- 状态色只用于 waiting、running、success、warning、failed。
- 控件小圆角，按钮克制。
- 不做 hero 区、装饰渐变、大型营销卡片或无必要装饰。

## 数据持久化

第一版配置使用本地文件保存：

```text
.config/providers.json
.config/model-routing.json
.config/github.json
.config/templates.json
```

任务保存在：

```text
output/jobs/<job-id>/
```

每个任务保存不可变快照，包括候选项目、已选项目、模板参数、生成脚本和生成分镜。用户编辑口播后，在渲染前写入更新后的脚本快照。

## 错误处理

- 缺少 GitHub Token 是警告，不是硬错误。
- GitHub API 触发限流时，应尽量显示重置时间。
- 缺少模型路由时，只阻塞需要该模型的阶段。
- 模型 API 失败时，显示供应商、模型、任务类型和错误摘要。
- AI 响应 JSON 解析失败时，把原始响应保存在任务目录里，便于检查。
- 渲染失败时，显示失败阶段和最后几行相关日志。
- 用户修改配置后，在已保存状态允许的情况下，可以从失败阶段重跑。

## 测试与验收

第一版验证应覆盖：

- 供应商配置校验和连接测试行为。
- 有 GitHub Token 和无 GitHub Token 两种候选项目查询。
- 候选快照持久化。
- 每个 AI 任务类型的模型路由查找。
- 草稿阶段和出片阶段的任务状态流转。
- 流水线适配层生成的文件符合当前 pipeline 预期。
- 现有 `--from-plan` 渲染路径仍可运行。
- 带编号输出文件名生成，包括同一天编号递增。
- 本地工作台 UI 冒烟测试：创建草稿、选择项目、编辑脚本、启动渲染、查看日志和最终产物。

## 后续增强

- 完整渲染前生成静态分镜预览。
- AI 脚本质检：检查夸大表达、模糊事实、节奏和可读性。
- 同一任务支持 v1、v2、v3 等产物版本。
- 模板库：模板预览图、版本说明和适用场景。
- 自动生成封面帧。
- 选题池和排期队列，用于每日或每周自动生成待确认草稿。
- GitHub 截图、README 内容和仓库元数据缓存。
- 单个 GitHub 项目推荐生产线。
- 科普类视频生产线。
- 发布辅助：标题、简介、话题标签和封面文案。
- 内部质量评分：口播质量、视觉可读性、可信度和发布准备度。
