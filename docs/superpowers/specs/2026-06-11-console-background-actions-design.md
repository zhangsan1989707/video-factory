# 控制台耗时步骤后台化设计

## 目标

把控制台热榜任务中仍然同步执行的耗时步骤统一改为后台任务，让 HTTP 请求快速返回，前端通过轮询继续展示进度。

## 范围

只处理 `P0-2 统一耗时步骤为后台任务` 的当前热榜主流程：

- 生成候选 `candidates`
- 重新生成候选 `regenerate-candidates`
- 确认项目并生成口播 `selection`
- 重新生成口播 `regenerate-script`
- 确认口播与质检 `script`
- 生成计划 `prepare-plan`
- 校验计划 `validate-plan`

最终渲染 `render-video` 已经是后台任务，本次沿用现有机制，不重做。

## 当前问题

这些步骤虽然内部有较明显的 IO 或模型耗时，但接口仍在请求线程里同步执行：

- `server.py` 直接 `asyncio.run(...)` 或直接调用同步函数。
- 前端默认等待完整返回体，然后再刷新详情。

这会带来两个问题：

1. 模型或网络慢时，HTTP 请求长时间挂起。
2. 用户刷新页面前后，对“任务仍在运行”这件事没有统一感知。

## 设计

### 服务端

- 复用现有 `start_async_job(...)` 机制。
- 为上述步骤增加统一的“后台启动包装层”：
  - 先做轻量同步校验，保留明确的 `400/404` 行为。
  - 再启动后台 worker。
  - 立即返回：
    - `started`
    - `active`
    - `job`
- 后台 worker 内部继续调用现有 `jobs.py` 逻辑，不改核心业务顺序。

### 失败处理

- 若后台 worker 抛错，统一记录日志并把任务标记为：
  - `status=failed`
  - `failed_stage=当前 stage`
  - `error=异常消息`
- 控制台重启后，现有 `reconcile_running_job()` 继续把悬挂 `running` 任务标记为失败，可重试。

### 前端

- 对上述步骤统一按“后台启动成功”处理：
  - 不再依赖完整结果体里的 `candidates / segments / artifacts`
  - 立即切到进度页
  - 调用 `startPollingCurrentJob()`
  - 由 `/api/jobs/{id}` 轮询刷新真实结果

### 验收标准

- 点击候选生成、重新生成候选、确认项目、重新生成口播、确认口播、生成计划、校验计划后，HTTP 快速返回。
- 这些步骤运行期间，详情页能看到 `status=running` 和对应阶段。
- 刷新页面后，仍能通过任务详情看到运行中的状态。
- 控制台重启或后台线程消失后，悬挂 `running` 任务会被标记为失败，可重试。
- 对应服务端和前端测试通过。

## 不做范围

- 不引入队列系统或外部任务服务。
- 不改变任务数据结构。
- 不处理 CLI 能力统一入控制台的问题。
