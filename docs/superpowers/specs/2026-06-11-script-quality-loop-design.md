# 2026-06-11 脚本质检闭环设计

## 背景

当前控制台已经能在确认口播时执行脚本质检，并在存在风险时阻断继续出片；但风险提示仍是平铺文本，用户需要自己在整段口播里找问题，闭环不够可操作。

## 本次范围

只补齐 `P1-2` 直接需要的最小闭环：

1. 后端在 `quality_report.json` 中新增结构化 `issues`，为风险项附带 `segment_id`。
2. 保留 `risk_flags`、`factual_notes`、`overclaim_notes` 旧字段，避免现有读取逻辑失效。
3. 前端把风险项渲染为可点击的“定位段落”按钮，点击后聚焦对应 textarea。
4. 对定位到的段落做短暂高亮，帮助用户确认修改位置。

## 设计取舍

- 不做自动改写。
  - 原因：这一步最容易越界覆盖用户已手改的文案，也会扩大 prompt、状态管理和回滚复杂度。
- 不额外引入新的接口。
  - 原因：当前 `/script` 确认口播后本来就会重新执行质检，复用现有保存链路即可满足“改后再检”。
- 风险定位优先使用确定性信息。
  - 本地规则风险直接绑定到 `project-{n}`。
  - 模型返回的风险先按项目名匹配，再尝试和段落文本做模糊匹配；匹配不上就保留未定位状态。

## 验收点

1. 质检失败时，`quality_report.json` 中能看到 `issues[].segment_id`。
2. 控制台风险表点击“定位段落”后，会聚焦到对应口播 textarea。
3. 用户修改后再次确认口播，会重新生成新的质检结果。
4. 手动忽略风险后，`quality_report.json` 仍保留忽略结果并允许继续出片。
